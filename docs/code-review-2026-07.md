# 📋 Final report

## Verdict

**ytm-native is a strong, well-thought-out project substantially above the average level of hobby code, but without a single safety net (git/linter/tests/CI) and with three systemic weaknesses that have already started to break behavior.** The architecture is mature: disciplined layering (raw InnerTube + parsers → Zustand stores → UI), a well-designed bridge between windows, pervasive "why" comments, deliberate race guards. The problems cluster not in "bad code" but in a few root causes — fixing them is local and pays off many times over.

Both static checks are green (`tsc --noEmit`, `cargo check`). The codebase is ~17k lines (TS + Rust).

## Strengths (do not rewrite)

- **File/process boundaries in Rust** are done carefully: yt-dlp via an argument array without a shell (no command injection), video_id/name sanitization (no path traversal), DPAPI cookie encryption on Windows, magic-byte MIME sniffing, a 32 KB floor against storyboard stubs.
- **Race guards where they are usually forgotten**: resolve token during stream resolution, StrictMode-safe unsubscription of Tauri listeners, `partialize` that never auto-plays on launch, versioned store migrations.
- **Defensive InnerTube parsing**: optional chaining, tree-walk with a WeakSet guard against cycles, token-echo and a 200-page cap in pagination, excellent comments about YT response quirks.
- **Thumbnail fallback chain** (iTunes → disk cache → largest variant, with onError demotion and cancellation) — genuinely thorough.

## Root causes (fix here — closes many findings at once)

1. **State synchronization between windows** — the biggest architectural weakness. The floating window is a separate JS context, shares localStorage keys with the main one on a last-writer-wins basis, and the close event does not distinguish user intent from a programmatic close. → high #7, #8; medium #5, #6, #7, #22; low layout.ts. **Fix direction:** floating window without a persist cache; a full set of actions in the bridge; a user-vs-programmatic close discriminator.
2. **Query-key drift** — ad-hoc string keys are scattered across 15+ files and have already diverged (dead invalidations, dead persist-whitelist, duplicate liked cache). → high #9; medium #13, #24; low query-client. **Fix:** centralize keys into a single factory module.
3. **Error semantics collapsed into "no data"** — `null` means both "empty" and "failure"; HTTP 200 is treated as success; a network failure is cached as "no lyrics" for an hour. → medium #18, #21; low musixmatch, stream.ts:79. And the mirror extreme — high #10, where a continuation error wipes what was already loaded. **Fix:** distinguish "empty"/"error", check the response status, do not drop the loaded UI over a load-more failure.
4. **store→engine contract via derived strings** — track transitions are passed as a videoId string and a boolean `playing`, not as explicit events/tokens; some async paths lack a staleness guard. → high #5, #6; medium #14, #15. **Fix:** a monotonic playToken per transition; a guard on stale async results.
5. **yt-dlp lifecycle / downloads-map races in Rust** — no timeout/kill, a write error leaves success=true, check-then-insert is not atomic. → medium #2, #3, #4; low #1365. **Fix:** timeout+kill, `ok=false` on a write error, an atomic get-or-spawn.
6. **Network trust boundary** — a local proxy without auth (CSRF/DNS-rebinding), SSRF in cache_cover. → high #1; medium #1. **Fix:** Host check + per-launch token; host allowlist + denial of private IPs/redirects.
7. **No safety net** (meta cause) — without git/tests/CI, none of the above has anything to catch a regression. → high #11.

- Strengths: the stream server correctly binds to the 127.0.0.1 loopback (not 0.0.0.0) on a random port; video_id and cover filename inputs are tightly sanitized (alnum/_/- , length caps, explicit ".." rejection) so path traversal is well-guarded; all external processes (yt-dlp) are spawned with argument arrays and no shell, so there is no command injection; cookies are encrypted with DPAPI on Windows and yt-dlp is deliberately never handed cookies; the Tauri capability http scope is host-limited and the CSP is otherwise restrictive (script-src 'self', object-src/frame-ancestors/form-action 'none'). Weaknesses: the local HTTP server pairs CorsLayer::permissive() with a complete absence of Origin/Host/auth validation on endpoints that spawn processes and write unbounded amounts to disk, making it exposed to CSRF/DNS-rebinding abuse; cache_cover is an unrestricted server-side URL fetch (SSRF primitive); cookie storage silently degrades to plaintext off-Windows; and raw auth cookies are handed to the webview while unsafe-headers is enabled, so any webview compromise yields full Google credentials. Overall the file system / process boundaries are handled carefully, but the network-facing trust boundary (who may talk to the local server, and what URLs it may fetch) is the weak point.

- The Rust backend is well above hobby-project average: thorough doc comments explaining non-obvious decisions (moov-atom streaming rationale, cookie-domain normalization, event dedup), consistent String-error handling with essentially no unwraps on untrusted data, correct input sanitization on both proxy routes (video-id and cover-filename validation rule out traversal), and thoughtful touches like DPAPI cookie encryption, magic-byte MIME sniffing, and the 32 KB stub-download floor. Its weaknesses cluster in concurrency and process lifecycle: check-then-act races on the downloads map and accounts index, no timeout/kill discipline on yt-dlp children, one blocking subprocess call on the main thread, and a completely unauthenticated permissive-CORS localhost server. Nothing here is architecturally unsound — the fixes are all local — but the absence of tests means none of these race/failure paths has ever been exercised deliberately.

The two systemic weaknesses are error handling that collapses every failure into null (defeating retry and caching \"no lyrics\" answers that are really network errors) and trust in remote search relevance without validating the match, which for Genius produces confidently displayed wrong-song lyrics. The time-sync highlight math in lyrics-view is correct at boundaries (linear scan, not binary search, but fine at lyric scale); its real defect is scroll-state lifecycle (no centering on mount, no reset on track change) rather than the sync logic itself.
